# ADR 0001 — identity-service объединяет auth и user

- **Статус:** Accepted
- **Дата:** 2026-06-26
- **Контекст этапа:** MVP-ядро готово (7 доменных сервисов + gateway).

## Контекст

В исходном наброске архитектуры предполагались два отдельных сервиса —
`auth-service` (аутентификация, выдача токенов) и `user-service` (профили,
связи teacher↔student). В реализации они объединены в один **identity-service**
с одной БД `identity_db`:

```
users, teacher_profiles, student_profiles, teacher_student_links
```

`identity` отвечает и за auth (подпись JWT, пароли PBKDF2, смена пароля), и за
пользовательские данные (профили, связи), и за канонический
`CheckTeacherStudentAccess`.

## Решение

Держать auth и user в одном сервисе `identity-service`. Не разделять на этом этапе.

## Обоснование

1. **Сильная связность данных.** Аутентификация, профиль и связь teacher↔student
   опираются на одни и те же сущности (`users`, `teacher_student_links`). Разделение
   привело бы к постоянным кросс-сервисным вызовам между auth и user на горячем пути.
2. **Один эмитент токенов и один источник доступа.** JWT и `CheckTeacherStudentAccess`
   логично держать рядом — обе операции читают `users`/links; иначе auth-service
   пришлось бы ходить в user-service на каждый access-check.
3. **microservices-lite.** Цель проекта — чёткие границы без преждевременной тяжести.
   Лишний сервис = ещё одна БД, миграции, контракт, деплой и сетевой хоп без выгоды
   для MVP.
4. **Граница уже правильная.** identity — лист графа вызовов (никого не зовёт), его БД
   изолирована, наружу он не торчит. Внутреннее объединение auth+user не нарушает
   правил «своя БД на сервис» и «доступ только через сервисный API».

## Последствия

**Плюсы:** меньше сетевых вызовов и кода, проще деплой и тесты, access-check дешёвый
(локальное чтение в identity), нет распределённой транзакции между auth и user.

**Минусы / на что обратить внимание:** identity несёт две ответственности — при росте
auth-фич (refresh/revoke, OAuth/SSO, MFA, audit сессий) сервис может раздуться; масштабировать
auth и user независимо нельзя; зона ответственности шире, чем у других сервисов.

## Когда пересматривать (триггеры для разделения)

- auth обрастает независимым стейтом и нагрузкой (сессии/refresh-store, OAuth, MFA),
  который хочется масштабировать/деплоить отдельно;
- появляется отдельная команда-владелец auth;
- нужно несколько способов входа / внешних identity-провайдеров;
- требования безопасности диктуют изоляцию секретов и токен-стора от пользовательских данных.

До наступления этих условий разделение даёт много работы и мало пользы — оставляем
объединённым.

## Альтернативы

- **Отдельные auth-service + user-service** (исходный набросок) — отклонено: кросс-сервисные
  вызовы на каждом access-check, лишняя инфраструктура без пользы для MVP.
- **Монолитный identity внутри gateway** — отклонено: нарушает «gateway тонкий» и
  «своя БД на сервис».
