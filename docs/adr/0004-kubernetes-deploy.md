# ADR 0004 — Kubernetes-деплой (kind) как альтернатива compose

- **Статус:** Accepted (реализация — Этап 8.6 roadmap)
- **Дата:** 2026-07-09
- **Контекст этапа:** production — один VM (4 GB + swap), Docker Compose +
  Caddy + GHCR, CI/CD через GitHub Actions (Этап 5K.1).

## Контекст

Проект хочет продемонстрировать оркестрацию Kubernetes: декларативный деплой,
self-healing, реплики и автоскейлинг, разделение liveness/readiness. При этом
production-VM мал (Kafka + Postgres + MinIO + 10 сервисов уже занимают почти
всю память), а демо на `netwatch-arsen-demo.ru` должно продолжать работать.

## Решение

1. **k8s-манифесты живут в репозитории (`deploy/k8s/`), запускаются локально
   в kind.** Production остаётся на compose. k8s — полноценный, проверяемый
   способ поднять весь стек, а не «мёртвый YAML»: DoD этапа — `smoke_mvp.py`
   зелёный против Ingress в kind.
2. **kustomize `base` + `overlays/local`** (без Helm): base описывает
   Deployments/Services/Jobs/конфиги, overlay — локальные образы, число
   реплик, ресурсы.
3. **Форма ресурсов:**
   - stateless-сервисы (gateway, 9 доменных) — `Deployment` + ClusterIP
     `Service`; доменные сервисы наружу не публикуются (правило №1 проекта
     сохраняется: только Ingress → gateway и realtime `/ws`);
   - gRPC-балансировка — headless Service + `dns:///` (round_robin между
     подами);
   - инфраструктура (postgres ×2 c учётом шарда chat, kafka, redis, minio) —
     `StatefulSet` + PVC; в реальном проде это были бы managed-сервисы,
     зафиксировано ниже;
   - `migrator` и `kafka-init` — `Job`'ы (идемпотентны уже сейчас);
   - вход — ingress-nginx: `/ws` → realtime, API-префиксы → gateway,
     остальное → frontend (переносим правила Caddyfile);
   - probes: liveness = `/health` (лёгкий), readiness = `/ready` (Этап 8.2) —
     под с недоступной БД выводится из балансировки, но не перезапускается
     зря;
   - HPA для notification-service по CPU (metrics-server в kind) — витрина
     автоскейлинга;
   - секреты — обычные k8s `Secret` (без внешних vault'ов).
4. **Образы** — те же GHCR-образы, что для compose (или локальная сборка +
   `kind load docker-image`); отдельных k8s-образов нет.

## Обоснование

1. **kind, а не k3s на prod-VM:** на 4 GB k3s + Kafka + Postgres + 10 сервисов
   не помещаются без деградации; эксперимент на живом демо рискует уронить
   его надолго. Локальный кластер даёт те же артефакты (манифесты, probes,
   HPA) без риска.
2. **kustomize, а не Helm:** один потребитель манифестов, шаблонизация не
   нужна; kustomize входит в kubectl и не добавляет зависимостей
   (правило «не плодить зависимости»).
3. **Инфраструктура в кластере — осознанное демо-упрощение:** цель — показать
   оркестрацию приложения; production-паттерн (managed PG/Kafka вне кластера)
   зафиксирован здесь, чтобы решение читалось как осознанный trade-off, а не
   незнание.
4. **Readiness из Этапа 8.2 — предусловие:** без `/ready` k8s наливал бы
   трафик в поды с мёртвой БД; сначала 8.2, потом 8.6.

## Последствия

**Плюсы:** в репо появляется рабочий, проверяемый k8s-деплой (self-healing,
реплики, HPA, rolling update); манифесты переносимы на любой managed-кластер
(поменять overlay: образы, storage class, ingress class, внешние DSN);
разделение health/ready наконец используется по назначению.

**Минусы / на что смотреть:**
- два способа деплоя = двойное сопровождение конфигурации (env, порты,
  зависимости) — снижаем риск тем, что источник правды по env остаётся
  `.env.example`, а k8s-конфиги генерируются из тех же значений;
- kind не показывает многонодовые сценарии планировщика (anti-affinity,
  зональность) — вне скоупа;
- TLS в локальном ingress не делаем (Caddy-функциональность остаётся
  prod-специфичной).

## Альтернативы

- **k3s/microk8s на prod-VM** — отклонено: не хватает ресурсов, риск для
  живого демо; вернёмся при апгрейде VM или втором сервере.
- **Helm-чарт** — отклонено: шаблонизация без второго потребителя не
  окупается; kustomize проще ревьюить.
- **Docker Swarm** — отклонено: целевая компетенция — именно Kubernetes.
- **Только манифесты без запуска** — отклонено: непроверенный YAML быстро
  протухает и ничего не доказывает.

## Когда пересматривать

- переезд на VM ≥8 GB или облачный k8s → продвигать k8s из local-overlay в
  production-overlay (ingress + cert-manager вместо Caddy, managed PG/Kafka);
- рост числа окружений (staging) → рассмотреть Helm/ArgoCD;
- необходимость zero-downtime миграций БД → Jobs + pre-deploy hooks.
