# ADR 0003 — Горизонтальное масштабирование сервисов и Kafka

- **Статус:** Accepted (реализация — Этапы 8.1, 8.3, 8.4 roadmap)
- **Дата:** 2026-07-09
- **Контекст этапа:** все сервисы в одном экземпляре; Kafka — один брокер,
  топики создаются auto-create'ом с 1 партицией.

## Контекст

Хотим запускать несколько реплик сервиса (первый кандидат —
notification-service: он и Kafka-consumer, и gRPC-бэкенд gateway). HTTP/gRPC
слой сервисов stateless — реплицируется свободно. Проблемы создают фоновые
компоненты:

1. **Outbox publisher** (`libs/events`, общий для lesson/assignment/finance/
   notification/chat): каждый тик читает `outbox_events WHERE status='pending'`
   **без блокировок**. Две реплики публиковали бы события дважды, а главное —
   могли бы нарушить **порядок** событий одного агрегата: реплика A забирает
   старый батч с `lesson.completed`, реплика B успевает опубликовать более
   поздний `lesson.cancelled` раньше. Consumers идемпотентны к дублям, но не
   к перестановке взаимоисключающих фактов.
2. **Топик на каждый event_type ломает порядок между типами.** Сейчас топик =
   `tutorflow.<event_type>` (`tutorflow.lesson.completed`,
   `tutorflow.lesson.cancelled`, …). Kafka гарантирует порядок только внутри
   партиции ОДНОГО топика — события одного агрегата разных типов идут в разные
   топики, и консьюмер может получить их в любом порядке, даже если outbox
   опубликовал их строго по порядку. Т.е. leader-lock (п.1) сам по себе не
   доводит гарантию порядка до консьюмера.
3. **Kafka-consumer**: consumer group уже настроен
   (`group_id: notification-domain-events`), но топики авто-создаются с
   1 партицией — вторая реплика в группе будет простаивать, параллелизма нет.
4. **Kafka-брокер один**: RF=1, падение брокера останавливает всю событийную
   шину; нельзя продемонстрировать отказоустойчивость.
5. `assignment` deadline-worker уже готов к репликам (`FOR UPDATE SKIP LOCKED`);
   там порядок не важен — образец другого паттерна.

## Решение

1. **Outbox: leader-lock через `pg_try_advisory_xact_lock`.** Тик publisher'а
   оборачивается в транзакцию, первым statement'ом берётся try-advisory-лок на
   константный ключ. Получила лок одна реплика — она публикует батч и
   помечает строки; остальные молча пропускают тик. Лок транзакционный:
   упавший лидер отпускает его автоматически (rollback), «застрять» нельзя.
   Лок живёт в БД сервиса → сервисы друг с другом не конкурируют.
2. **Топики по доменам: `tutorflow.<domain>.events`.** Пять топиков —
   `tutorflow.lesson.events`, `tutorflow.assignment.events`,
   `tutorflow.finance.events`, `tutorflow.chat.events`,
   `tutorflow.notification.events`; `event_type` остаётся полем envelope
   (там он и так есть). Все события одного агрегата (`lesson_id` как ключ)
   попадают в одну партицию одного топика → порядок
   `lesson.completed` → `lesson.cancelled` гарантирован до консьюмера.
   Диспатч консьюмеров уже сделан по `envelope.event_type`, а не по имени
   топика (проверено по коду notification) — меняются только
   `TopicForEventType` в libs/events (маппинг `event_type` → домен),
   списки топиков в static_config'ах консьюмеров и kafka-init.
3. **Явное создание топиков.** One-shot контейнер `kafka-init` (аналог
   `migrator`) создаёт 5 доменных топиков: `partitions=3`, RF=1 в dev /
   RF=3 в multi-broker профиле. Auto-create остаётся страховкой.
4. **Consumer scaling.** Реплики сервиса входят в общий consumer group и делят
   партиции (стандартный Kafka-механизм). Дедупликация — существующий inbox
   (`processed_events`). notification-service ×2:
   `docker compose up --scale notification-service=2` (или `deploy.replicas`,
   если проверено, что наша версия compose его честно исполняет);
   в k8s — `replicas: 2` + HPA (ADR 0004).
5. **Multi-broker Kafka — отдельным compose-профилем** (`docker-compose.scale.yml`):
   `kafka-1..3`, KRaft-кворум из трёх, `min.insync.replicas=2`, топики RF=3.
   Producer уже `enable_idempotence: true` — librdkafka при идемпотентности
   принудительно пишет с `acks=all`, иначе RF=3/min.insync=2 не давали бы
   реальной гарантии; фиксируем `acks=all` явно.
   Dev и production по умолчанию остаются на одном брокере (ресурсы VM).
6. **Брокеры конфигурируются через env** (`KAFKA_BROKERS`, по умолчанию
   `kafka:9092`) вместо захардкоженного `secdist.json` — один образ работает
   с любым составом кластера.

## Обоснование (ключевой выбор: advisory lock vs SKIP LOCKED)

`FOR UPDATE SKIP LOCKED` — первый напрашивающийся вариант (параллельная
публикация несколькими репликами), но он **ломает порядок**: захват строк не
атомарен по агрегату, две реплики могут публиковать события одного агрегата
одновременно в произвольном порядке. Финансовый флоу
(`lesson.completed` → charge, `lesson.cancelled` → correction) на перестановку
не рассчитан.

Пропускная способность outbox не является узким местом (батч 100 строк на тик
на сервис при ручном темпе домена), поэтому параллелизм публикации не нужен —
нужна только безопасность реплик. Leader-lock даёт её тремя строками кода,
сохраняя строгий порядок `ORDER BY created_at, id` одного писателя.

SKIP LOCKED остаётся правильным инструментом там, где порядок не важен —
как в deadline-worker'е assignment.

**Семантика доставки — честная формулировка.** Система даёт
`at-least-once delivery + идемпотентный консьюмер = effectively-once эффект в
БД`, а НЕ «exactly once»: событие может прийти повторно (replay, повтор батча
после падения лидера), но inbox/unique-констрейнты не дают второго эффекта.
В документации и DoD не употреблять «ровно один раз» про доставку. Отдельное
следствие: когда notification-service начнёт слать внешние side-effects
(email/Telegram — вне MVP), «отправить письмо» нельзя делать прямо в
консьюмере — понадобится собственный outbox notification-service, иначе письмо
может уйти дважды.

## Последствия

**Плюсы:** любую службу можно запускать в N реплик без дублей и перестановок;
consumer-параллелизм включается числом партиций; шина переживает падение
брокера (в scale-профиле); publisher-код общий — фикс в `libs/events`
покрывает все 5 сервисов с outbox.

**Минусы / на что смотреть:**
- publisher фактически singleton per-БД: масштабирование публикации —
  вертикальное (батч/период). При реальном росте — SKIP LOCKED с группировкой
  по `aggregate_id` (партиционирование публикации), зафиксировано как триггер
  пересмотра;
- тик стал транзакционным: упавший в середине батча лидер повторит **весь**
  батч (раньше — только неопубликованный хвост). Дублей чуть больше в редком
  сценарии — consumers идемпотентны, приемлемо;
- Kafka `Publish` выполняется ВНУТРИ открытой БД-транзакции: при деградации
  Kafka транзакция (и advisory-лок) держится дольше обычного. Ограничители:
  батч ≤100, `delivery_timeout: 5s` у producer'а, период тика; добавить лог
  «tick took too long» и метрику длительности тика (8.7). Для demo-нагрузок
  приемлемо; при росте — вынести Publish за транзакцию (claim → publish →
  mark в отдельных шагах);
- переход на доменные топики — смена контракта шины: старые
  `tutorflow.<event_type>` топики дренируются и удаляются (dev — пересоздание;
  prod-demo — короткий даунтайм консьюмеров при переключении допустим);
- консьюмер доменного топика получает и «не свои» event_type (например,
  realtime читает весь `chat.events`, а нужен только `message.read`) —
  фильтрация по `event_type` уже есть в диспатче, лишние события дёшево
  пропускаются (уже сейчас код логирует unsupported и выходит);
- gRPC к реплицированному сервису балансируется Docker DNS (per-connection);
  честный round_robin появится в k8s через headless Service (ADR 0004);
- `partitions=3` фиксирует нижнюю границу параллелизма группы — больше 3
  консьюмеров на топик смысла не имеют (достаточно для демо).

## Альтернативы

- **Topic-per-event_type (текущая схема)** — отклонено: не даёт порядка между
  событиями разных типов одного агрегата (главный мотив всей цепочки
  ordering-гарантий). Плюс 17 топиков вместо 5 — больше конфигурации без выгоды.
- **Один общий топик `tutorflow.events`** — отклонено: все консьюмеры читали бы
  весь поток; домен — естественная граница подписки и владения (producer
  домена → его топик).
- **`FOR UPDATE SKIP LOCKED` в outbox** — отклонено для publisher'а из-за
  нарушения порядка (см. выше).
- **Отдельный сервис-publisher (CDC/Debezium)** — отклонено: новая тяжёлая
  инфраструктура ради проблемы, решаемой одним lock'ом; противоречит
  microservices-lite.
- **Kafka transactions / exactly-once** — отклонено: сложность несоразмерна;
  at-least-once + inbox уже даёт нужные гарантии.
- **Session-level advisory lock (`pg_advisory_lock`)** — отклонено: требует
  аккуратного освобождения; xact-вариант самоочищается.

## Когда пересматривать

- outbox-lag растёт (одного писателя мало) → публикация с параллелизмом по
  агрегатам (SKIP LOCKED + группировка) или CDC;
- потребители не успевают при 3 партициях → пересоздать топики с бОльшим
  числом партиций (offline, dev-данные);
- события станут критичными к потере — вынести Kafka на managed-кластер,
  DLQ/retry-топики (backlog «Observability 5K»).
