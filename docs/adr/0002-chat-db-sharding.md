# ADR 0002 — Шардирование chat_db по dialog_id

- **Статус:** Accepted (реализация — Этап 8.5 roadmap)
- **Дата:** 2026-07-09
- **Контекст этапа:** backend стабилен (Этапы 1–6), production demo на compose.

## Контекст

`chat_db` — самая быстрорастущая БД проекта: `messages` растёт с каждым
сообщением каждой пары teacher↔student, к ней же привязаны
`message_attachments` и `read_markers`. Все остальные БД растут медленно
(пользователи, занятия, чеки). Одна PostgreSQL-нода ограничивает и объём, и
пропускную способность записи чата.

Задача этапа — продемонстрировать горизонтальное масштабирование БД
(шардирование) на реальном сервисе, не нарушая правил проекта: каждый сервис
владеет своей БД, чужие БД не читает, роутинг — забота сервиса-владельца.

Особенность схемы: все «горячие» запросы чата привязаны к одному диалогу
(`ListMessages`, `SendMessage`, `MarkRead`, unread count). Единственный
кросс-диалоговый запрос — `ListDialogsForUser` (список диалогов пользователя
с last_message и unread_count).

## Решение

1. **Ключ шардирования — `dialog_id`, hash-based.** Шардов N=2 (список DSN в
   `CHAT_DATABASE_URLS`, число фиксируется конфигурацией). Функция:
   FNV-1a от нормализованной (lowercase, без дефисов) строки UUID, mod N —
   стабильна между рестартами, репликами и языками (используется и в
   `scripts/reshard_chat.py`).
2. **Ко-локация.** На шарде диалога живут ВСЕ его данные: строка `dialogs`,
   `messages`, `message_attachments`, `read_markers` и `outbox_events` этого
   диалога. FK внутри шарда сохраняются, транзакция `SendMessage`
   (message + attachments + last_message_at + outbox) остаётся локальной
   одному шарду — распределённых транзакций нет.
3. **Детерминированный `dialog_id` = UUIDv5(namespace, teacher_id ‖ student_id).**
   Диалог пары вычислим без поиска: find-or-create роутится сразу на нужный
   шард, глобальный `UNIQUE (teacher_id, student_id)` больше не нужен —
   уникальность пары гарантирует сам id (вставка идемпотентна
   `ON CONFLICT (id) DO NOTHING`). Это убирает единственную причину для
   глобального индекса/каталога.
4. **`ListDialogsForUser` — scatter-gather.** Запрос выполняется на каждом
   шарде (там же локально считаются unread_count и last_message — JOIN'ы
   остаются пошардовыми), результаты сливаются в сервисе по
   `last_message_at DESC`. Диалогов у пользователя десятки — merge дёшев.
5. **Outbox — на каждом шарде.** `message.sent`/`message.read` пишутся в
   `outbox_events` того же шарда той же транзакцией; publisher chat-service
   обходит все шарды, leader-lock (ADR 0003) берётся на каждом шарде
   независимо. Порядок событий per-dialog сохраняется: диалог всегда на одном
   шарде, ключ Kafka-партиции — `dialog_id`.
6. **Миграция существующих данных** — `scripts/reshard_chat.py`: пересчитывает
   новые детерминированные id старых диалогов, переносит строки на целевые
   шарды. Объёмы demo-данных позволяют оффлайн-прогон (downtime chat-service
   на время миграции приемлем).

## Обоснование

1. **dialog_id — естественная единица локальности.** Все горячие запросы —
   в пределах диалога; ко-локация делает их single-shard, сохраняет FK и
   локальные транзакции.
2. **UUIDv5 убивает главный минус hash-шардинга.** Без него find-or-create по
   паре (teacher, student) требовал бы scatter-gather + глобальной уникальности
   (гонка двух create → два диалога на разных шардах). Детерминированный id
   решает и роутинг, и уникальность одним механизмом.
3. **Соответствует духу проекта.** Роутинг — маленький `ShardRouter` внутри
   репозитория chat-service; снаружи контракт не меняется, остальные сервисы
   и gateway не знают о шардах.
4. **Работает при N=1.** Один DSN в `CHAT_DATABASE_URLS` = текущее поведение;
   dev-окружение можно не усложнять, «шардированный» профиль — отдельно.

## Последствия

**Плюсы:** запись и объём чата масштабируются добавлением шардов; горячие
запросы остаются single-shard; нет распределённых транзакций и глобальных
индексов; механика шардирования видна в коде (цель pet-проекта).

**Минусы / на что смотреть:**
- смена числа шардов = ручной решардинг (`reshard_chat.py`), consistent
  hashing не делаем (вне скоупа, зафиксировано в Этапе 8);
- `ListDialogsForUser` — fan-out на все шарды: латентность = max по шардам,
  при большом N понадобится кэш/материализация (не актуально при N=2);
- миграция меняет существующие `dialog_id`: исторические payload'ы событий и
  старые in-app уведомления ссылаются на старые id (для demo-данных приемлемо;
  фронт dialog_id не персистит);
- один «мега-диалог» не разрезается (шард = диалог целиком) — для чата
  teacher↔student это не проблема по определению домена.

## Альтернативы

- **Directory-based (таблица dialog_id → shard в каталоге)** — отклонено:
  лишний lookup на каждом запросе, каталог становится общей точкой отказа и
  всё равно не решает гонку find-or-create без глобального UNIQUE.
- **Декларативное партиционирование PostgreSQL (PARTITION BY HASH)** —
  решает объём таблицы/индексов, но не масштабирует запись и железо: всё
  остаётся на одной ноде. Не демонстрирует целевую компетенцию.
- **Citus / шардинг-мидлвары** — отклонено: тяжёлая зависимость, скрывает
  механику; в учебном проекте ценность именно в ручной реализации роутинга.
- **Шардирование по teacher_id** — рассмотрено: делает `ListDialogsForUser`
  single-shard для учителя, но не для ученика, и создаёт горячие шарды
  (активный преподаватель = весь трафик на одном шарде). dialog_id ровнее.

## Когда пересматривать

- появился прод-трафик и N=2 мало → добавить шарды + прогнать решардинг;
  при частых решардингах — перейти на consistent hashing / directory;
- `ListDialogsForUser` стал узким местом при росте N → материализованный
  список диалогов пользователя (read-model) или кэш;
- появится второй кандидат на шардирование (например, `report_db`) →
  вынести `ShardRouter` в `libs/common` (сейчас осознанно НЕ выносим —
  правило «не плодить преждевременных абстракций»).
